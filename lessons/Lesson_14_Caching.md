# Phase 14: Caching

Right now, every time your workflow runs, `dotnet restore` reaches out to NuGet.org and downloads every single dependency from scratch. 
As your project grows and you add dozens of libraries (Entity Framework, Serilog, AutoMapper, etc.), this step will take longer and longer.

We can fix this by introducing **Caching**.

---

## What is Caching in GitHub Actions?

Caching allows you to save folders (like the downloaded NuGet packages) from one workflow run and instantly restore them in the next run. 

*   **Cache Miss:** The first time the workflow runs, the cache is empty. `dotnet restore` downloads everything from the internet. At the end of the job, GitHub saves these downloaded files.
*   **Cache Hit:** The next time the workflow runs, GitHub sees the cache exists. It instantly copies the saved files into the runner. `dotnet restore` runs, sees the files are already there, and finishes in milliseconds!

---

## The Cache Key

How does GitHub know if your cache is still valid? What if you added a *new* NuGet package? You wouldn't want to use the old cache.

We solve this using a **Cache Key**. The key is usually a unique string generated by hashing your project files. If you change a `.csproj` file (by adding a package), the hash changes, the key changes, and GitHub knows to create a fresh cache (a Cache Miss).

---

## Step 1: Update `ci.yml`

We will use the official `actions/cache@v4` action. We must place it **before** `dotnet restore`.

Update your `build-and-test` job steps to look like this:

```yaml
    steps:
    - name: Checkout Code
      uses: actions/checkout@v4

    - name: Setup .NET SDK
      uses: actions/setup-dotnet@v4
      with:
        dotnet-version: '10.0.x'

    - name: Cache NuGet Packages
      uses: actions/cache@v4
      with:
        path: ~/.nuget/packages
        key: ${{ runner.os }}-nuget-${{ hashFiles('**/*.csproj') }}
        restore-keys: |
          ${{ runner.os }}-nuget-

    - name: Restore Dependencies
      run: dotnet restore
```
*(Leave the build, test, publish, and upload steps exactly as they were).*

---

## Keyword Explanations

*   **`uses: actions/cache@v4`**: The official action to save and restore folders.
*   **`path: ~/.nuget/packages`**: This is the secret hidden folder on Linux (`~/.nuget/packages`) where .NET stores all downloaded packages. This is the folder we want to cache.
*   **`key: ${{ runner.os }}-nuget-${{ hashFiles('**/*.csproj') }}`**: This is the magic. 
    *   `runner.os` ensures a Linux cache is only used on a Linux runner.
    *   `hashFiles('**/*.csproj')` scans every `.csproj` in your repo and generates a unique fingerprint. If you add a package, this fingerprint changes, forcing a fresh download.
*   **`restore-keys:`**: If an exact cache hit isn't found, GitHub will fall back and try to find any older cache that starts with `Linux-nuget-`. This way, you at least get *most* of your packages instantly, and only have to download the new ones.

### End of Lesson 14

#### 1. Summary
Caching significantly speeds up CI pipelines by reusing downloaded dependencies across runs. We use a **Cache Key** based on a hash of our `.csproj` files to ensure we always get a fresh cache when dependencies change. `actions/cache` must run before the step that downloads the dependencies (`dotnet restore`).

#### 4. Small Exercise for You
1.  Add the `actions/cache` step to your `ci.yml` just before the Restore step.
2.  Push to `master`. The first run will be a **Cache Miss** (it will save the cache at the end of the job).
3.  Make a tiny change to a `.cs` file or `README.md` and push again.
4.  Look at the logs for the Cache step on the second run. You should see a message saying **Cache restored successfully** (a Cache Hit). Notice how much faster `dotnet restore` runs!

#### 5. Common Mistakes
*   Putting the cache step *after* the restore step. It's too late! The runner already downloaded everything from the internet.
*   Caching the wrong directory. If you cache the project workspace instead of the global `~/.nuget/packages` folder, it won't work correctly for .NET.

#### 6. Best Practices
*   Use `actions/setup-dotnet`'s built-in caching if you want a simpler setup. (Note: `setup-dotnet` actually has a `cache: true` option built-in nowadays, but manually configuring `actions/cache` as we did here teaches you the fundamental concept used across all languages!).
*   Monitor cache sizes. GitHub gives you 10GB of free cache storage per repository. Old caches are automatically evicted.
